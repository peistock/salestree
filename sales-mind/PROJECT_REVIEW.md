## 一、项目定位与核心假设

### 1.1 一句话定义
SalesMind 是一个**单家庭部署的 AI 管家系统**，通过企业微信与家庭成员交互，核心服务对象是 4 位老人，由夫妻担任管理员。

### 1.2 核心假设（这些假设构成了整个架构的基石）

| 假设 | 具体表述 | 若假设不成立的影响 |
|------|---------|------------------|
| **单家庭** | 系统只服务一个家庭，不考虑多租户 | 若未来商业化，需重写数据层和接入层 |
| **企微渠道** | 家庭成员使用企业微信（或外部群中的个人微信） | 若老人拒绝安装企微，整个接入层作废 |
| **Mac 服务器** | 服务运行在爸爸的 Mac 上，7×24 小时开机 | Mac 合盖/断电/网络波动会导致服务中断 |
| **Claude API** | 核心智能依赖 Claude API（Haiku/Sonnet） | API 涨价、限速、区域封锁会直接影响可用性 |
| **本地隐私优先** | 敏感数据（病历）本地处理，通用查询走云端 | 本地模型（Qwen2.5-14B）能力有限，可能影响医疗相关回答质量 |
| **夫妻录入** | 用药/日程等数据由夫妻手动录入，非老人主动维护 | 数据更新摩擦高，长期可能因懒惰而数据过时 |

## 二、架构设计（分层拆解）

### 2.1 整体数据流

```
家庭成员（企微/微信）
    │
    ▼ 文字/语音消息
┌─────────────────────────────────────────────┐
│ 接入层（WeChat Adapter）                      │
│ - XML 解密/验签（明文模式）                    │
│ - 消息去重（MsgId 内存缓存，10k 窗口）         │
│ - 5秒超时保护（立即回"收到"，后台异步）         │
└──────────────────┬──────────────────────────┘
                   │
    ┌──────────────┴──────────────┐
    ▼                             ▼
┌─────────────┐           ┌─────────────┐
│ 文字消息     │           │ 语音消息     │
│ 直接进入     │           │ AMR→FFmpeg  │
│ 记忆层       │           │ →WAV→ASR   │
└──────┬──────┘           └──────┬──────┘
       └─────────────┬─────────────┘
                     ▼
┌─────────────────────────────────────────────┐
│ 记忆层（Memory）🔥 四层设计                    │
│                                             │
│ L1 短期记忆：当前对话线程（20轮保留）           │
│   └─ 触发条件：消息数≥20 或 闲置>30分钟        │
│   └─ 动作：LLM 生成摘要 → 归档到中期记忆       │
│                                             │
│ L2 中期记忆：日/周摘要 + 历史线程摘要          │
│   └─ 召回：对话时自动加载最近 3 条摘要         │
│                                             │
│ L3 长期记忆：用户画像（偏好/健康）             │
│   └─ 自动沉淀：Agent Reflect 阶段提取关键词    │
│   └─ 存储：PostgreSQL user_profiles           │
│                                             │
│ L4 技能库：本地 Markdown 文件                │
│   └─ 自动生成：复杂任务成功后沉淀              │
│   └─ 当前限制：只按文件名匹配，无向量检索      │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│ 路由层（Model Router）                        │
│ - 简单问答 → Claude Haiku（低成本，快）         │
│ - 复杂分析 → Claude Sonnet（推理强）            │
│ - 医疗敏感 → 本地 Qwen2.5-14B（隐私）           │
│ - 知识检索 → BGE-small-zh + PGVector           │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│ Agent 执行引擎                                │
│ Plan → Act（工具调用）→ Observe → Reflect     │
│ - 最多 6 步 Act 循环，防无限循环               │
│ - Plan 阶段 LLM 判断 simple/complex           │
│ - Reflect 阶段生成老人友好回复                 │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│ 输出层（Reply）                               │
│ - 老人友好格式化（短句、亲切、每段≤30字）       │
│ - 企微主动推送（3次指数退避重试）              │
│ - 医疗免责声明自动追加                         │
└─────────────────────────────────────────────┘
```

## 三、关键设计决策与辩论点

### 3.1 决策一：自研而非基于 Dify/LangChain

**选择**：完全自研，不依赖低代码平台。

**理由**：
- 完全掌控数据流，敏感数据（病历）可严格本地化处理
- 企微渠道可深度定制（消息去重、5秒超时、语音特殊处理）
- 避免平台锁定，长期维护自主

**潜在风险**：
- 重复造轮子：Dify 已解决的多租户、可视化编排、版本管理都需要自己写
- 维护成本高：安全补丁、依赖升级、API 变更都需要自己跟进
- 人才依赖：只有爸爸能维护，其他人看不懂代码

**辩证结论**：单家庭场景下自研合理，但若想产品化，应迁移到 Dify/LangChain 底座。

---

### 3.2 决策二：PostgreSQL + PGVector 而非专用向量库

**选择**：用 PostgreSQL 扩展 PGVector 做向量检索，而非 Chroma/Milvus/Pinecone。

**理由**：
- 少维护一个服务，备份简单（pg_dump 一次搞定关系数据 + 向量）
- SQL 查询可 join 关系数据和向量数据
- 单家庭数据量小（<10万条），PGVector 性能足够

**潜在风险**：
- PGVector 的 ivfflat 索引在高维（>1000维）下性能下降，当前 512 维尚可
- 向量距离计算在 PG 里不如专用库优化，未来数据量增大后可能成为瓶颈
- 备份/恢复时向量索引需要重建

**辩证结论**：当前选择合理，若未来知识库 >10万片段，应评估迁移到专用向量库。

---

### 3.3 决策三：本地 Embedding 模型（BGE）而非云端 API

**选择**：本地运行 BGE-small-zh-v1.5 生成 Embedding。

**理由**：
- 病历/体检报告不上云，隐私合规
- 离线可用，不依赖网络
- 无额外 API 费用

**潜在风险**：
- 首次下载模型 ~100MB，需要网络
- Mac M 系列芯片可跑，Intel 老 Mac 可能较慢
- 模型更新需要自己跟进，云端 Embedding API 持续迭代

**辩证结论**：单家庭、隐私敏感场景下本地模型是必须的，但应定期评估模型版本（当前 bge-small-zh-v1.5 是 2023 年模型）。

---

### 3.4 决策四：Claude API 为主要智能层

**选择**：核心智能依赖 Claude API（Anthropic），而非国内大模型。

**理由**：
- Claude 在复杂推理、长上下文、指令遵循上表现优异
- Haiku 成本极低（$0.25/1M tokens），适合家庭高频对话
- Sonnet 在 Plan/Reflect 阶段推理可靠

**潜在风险**：
- 数据出境：对话内容发送至 Anthropic 美国服务器
- 区域风险：若政策变化可能导致 API 访问受限
- 成本波动：Anthropic 可能涨价
- 网络延迟：国内访问 Anthropic API 有时较慢

**缓解措施**：
- 医疗/敏感数据走本地 Qwen2.5-14B
- 日常查询走 Haiku（成本已很低）

**辩证结论**：当前选择合理，但应监控国内大模型（DeepSeek、Kimi、通义）的能力进展，准备备选路由。

---

### 3.5 决策五：企微而非飞书/个人微信

**选择**：企业微信自建应用。

**理由**：
- 官方 Bot API，稳定合法，不会封号
- 企业微信外部群可拉个人微信用户，老人无需装新 App
- 支持主动推送、定时消息

**潜在风险**：
- 个人组建团队无法登录 Web 管理后台（官方限制），必须使用营业执照注册"企业"
- Bot 不支持发原生语音，只能发文字或 MP3 文件
- 企微生态封闭，API 功能有限（无朋友圈、无小程序跳转）

**辩证结论**：企微在"稳定性"上最优，但"体验"不如飞书（飞书 Bot 支持原生语音、卡片消息）。若老人对企微接受度低，迁移到飞书的成本是重写 `mind/wechat.py`（约 200 行）。

## 七、辩证分析议题（供其他 AI 讨论）

### 议题 1：架构是否过度设计？
单家庭场景下，是否需要四层记忆、向量检索、Docker Compose？一个 300 行的 Python 脚本 + SQLite + 企微 Webhook 是否更务实？

### 议题 2：技术债风险评估
- 1734 行代码由 AI 生成，人类可读性如何？爸爸离职后谁维护？
- 依赖 14 个外部库，版本冲突如何处理？
- 没有单元测试，如何确保重构不破坏功能？

### 议题 3：数据隐私的"伪本地"问题
虽然病历走本地模型，但企微云端仍保存聊天记录（建议设 7 天清理）。真正的"本地优先"是否应放弃企微，改用完全离线的方案（如局域网 Web UI）？

### 议题 4：商业化可行性
若从单家庭扩展到 SaaS：
- 当前单进程 FastAPI 能否支撑 1000 家庭并发？
- 每个家庭的向量库是否应隔离？
- 计费模型按家庭数、消息数还是 token 数？

### 议题 5：老人体验 vs 技术复杂度
当前方案技术栈复杂（Docker、PostgreSQL、向量检索、LLM API），对非技术家庭成员的维护成本极高。是否应该牺牲部分"智能"换取"简单"？

### 议题 6：语音交互的权宜之计
企微 Bot 不能发原生语音，TTS 生成的 MP3 以文件形式发送，老人需要点击播放。这是否比纯文字更差？是否应换平台（飞书支持原生语音）？
